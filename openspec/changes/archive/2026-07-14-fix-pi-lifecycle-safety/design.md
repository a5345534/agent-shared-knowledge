## Context

The repository has two divergent lifecycle extension implementations: the packaged extension and a large TypeScript string generated by `scripts/hooks/pi_lifecycle.py`. The packaged version uses obsolete Pi APIs. Both versions write candidates into `ctx.cwd`, then invoke a hook that may apply, stage, and commit changes in that checkout. Pi package users can also run `knowledge-query init`, causing package and project-local lifecycle extensions to load together.

## Goals / Non-Goals

**Goals:**
- Compile and run against the supported Pi API.
- Keep a clean checkout unchanged under default lifecycle configuration.
- Permit explicit inbox materialization and adopter-provided external materializers.
- Separate absorption file changes from Git commit policy.
- Maintain one canonical lifecycle implementation and avoid duplicate loading.

**Non-Goals:**
- Implement a repository-specific worktree/PR materializer.
- Remove manual `plan` or `apply` CLI workflows.
- Guarantee atomic behavior for arbitrary adopter commands.

## Decisions

1. **Safe default is review-only.** Candidate extraction may run, but without an explicit materializer setting it only reports validated candidates and writes no repository files. This prioritizes checkout integrity over implicit persistence.
2. **Materializer modes are explicit.** `SHARED_KNOWLEDGE_MATERIALIZER=inbox` opts into legacy `knowledge/inbox` writes. `SHARED_KNOWLEDGE_MATERIALIZER=command` delegates validated candidate JSON to an argv configured as JSON in `SHARED_KNOWLEDGE_MATERIALIZER_COMMAND`. Invalid configuration fails closed without repository writes.
3. **Git policy is a hook option.** Add `--git-mode none|commit`; `none` never stages or commits. The extension uses `none` when the explicitly selected inbox materializer invokes absorption. Manual operators can explicitly request commit behavior.
4. **The packaged extension is canonical.** The generated project-local extension becomes a small re-export loader pointing to the canonical file in the submodule. Init detects a project-local Pi package declaration and skips installing a second extension.
5. **Use current Pi APIs.** Consume `event.preparation.messagesToSummarize`, `ctx.model`, model registry credentials, `convertToLlm` then `serializeConversation`, and `complete(model, context, options)`. Parse text content as JSON and use supported notification levels.
6. **Type checking is a release gate.** Add Pi peer dependencies, TypeScript development tooling, a no-emit config, and tests covering package manifest and generated-loader behavior.

## Risks / Trade-offs

- [Safe default no longer persists candidates automatically] → Clearly notify and document the explicit materializer modes.
- [External command is adopter-controlled code execution] → Require explicit mode plus JSON argv; do not invoke through a shell.
- [Project-local package detection may miss unusual settings composition] → Keep `--skip-hook` and idempotent installation available; document one-extension invariant.
- [No-git mode can still modify files when inbox absorption is explicitly enabled] → Name and document it as Git-policy separation, while keeping review-only as the safe default.

## Migration Plan

1. Update and type-check the canonical package extension.
2. Existing package adopters receive review-only behavior after update unless they explicitly configure materialization.
3. Submodule adopters rerun init; the generated extension becomes a canonical loader, or installation is skipped if the package is already declared.
4. Rollback by pinning the previous commit and disabling lifecycle auto-apply with `SHARED_MEMORY_ABSORB_AUTO_APPLY=0`.
